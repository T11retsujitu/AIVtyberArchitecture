# 04 — Perception Distortion Policy（知覚の歪み）★

このドキュメントは `docs/07`（Perception Schema）の**姉妹編**。
`docs/07` が**形**（スキーマ・生メカニクス禁止）を定めるのに対し、ここは**意味論**——`perceive()` の描写が RawState に対して**どこまで真実で、どこから夢として歪んでよいか**——を定める。全ゲームの境界条件であり、歪みを使う・使わないに関わらず perceive 実装前に読む。

スキーマ・契約・不変条件は**一切変えない**。歪みは既存の `description`/`summary` 等の語彙の中で行う。

---

## なぜ歪みが要るのか

期待ギャップエンジン（`docs/03 §1`）の条件②「AIちゃんが視聴者正解を外す理由」の**主レバー**が知覚の歪み。

- 視聴者には Tier B（`docs/11 §2`）で RawState の真実を見せる
- AIちゃんには歪んだ描写を渡す——**視聴者は知ってる、本人は知らない**（劇的アイロニー）

LLM の性能を下げたり、プロンプトで愚かさを演出したりする必要はない。**入力（描写）を歪めれば、賢いモデルでも筋の通った間違いをする**。そして「夢だから知覚が歪む」はキャラ設定（`docs/00 §1`：夢を歩いている）そのものであり、演出上の嘘にならない。

歪みは特別な新機能ではない。`perceive()` がもともと持つ「RawState→描写」の写像には解釈の裁量があり（praise-room の `closeness 2 →「やわらかい光に、つつまれている」`も既に弱い解釈を含む）、この doc はその裁量に**規範**を与える。規範なしにゲームごとに場当たりで嘘をつき始めるのが最悪の形——それを防ぐのが目的。

---

## 1. 原理 — 夢は「信頼できない語り手」、ただし決定論的

- 歪みは **RawState → AIChanPerception の純関数写像の一部**。乱数・時刻・状態外の情報を使わない（`docs/02` の決定論と同じ制約）。
- 同じ seed＋同じ action 列 → 同じ歪んだ perception 列。既存の replay 決定論テスト（`docs/07` 整合の検証）にそのまま乗る。
- 歪むのは**解釈**であって**機構**ではない。`apply`/`isTerminal` は歪みを知らず、真実の RawState だけで動く。

## 2. 歪めてよいもの（whitelist）

すべて `scene.summary`／`elements[].description`／`affordances[].label,hint`／`feedback.description` の**描写語彙レベル**で行う：

- **正体・意図のラベリング**：怪物 → 「重たい気配」、罠 → 「親切そうな窪み」、嘘つき → 「困っているみたいな声」。
- **危険/安全のトーン**：切迫した状況を、のどかに・遠くに描写する（逆も可）。
- **夢的な言い換え**：日常物を別の何かとして描く（信号機が喋る・机が眠たがる）。ダイジェティックな擬人化はこの一種。

## 3. 歪めてはいけないもの（hard rules）

| # | ルール | 破るとどうなるか |
|---|---|---|
| 1 | **affordances の実在性**：提示した action は今の state で必ず `apply()` が受理する（`docs/07` フィールド契約のまま）。嘘の選択肢・実行できない誘いを並べない | take が矛盾で壊れる。視聴者にも嘘の選択肢が見える |
| 2 | **質信号の方向は誠実**：`salience`（存在感の強弱）・`feedback.valence`（good/bad/neutral/strange の方向）・`closure`（局面）は RawState の真実に対して誠実に付ける。**歪みは description に住まわせ、質信号まで嘘をつかない** | LLM の行動選択が乱択化し、失敗の型の引き金（`docs/01 §3`＝valence/salience/closure が担う）が全部壊れる |
| 3 | **因果の非捏造**：起きていないことを起きたと描写しない。歪めてよいのは「何と**見える**か」であって「何が**起きた**か」ではない | 視聴者に嘘の因果が見える（take-integrity・`docs/12` と同根）。オチが捏造になる |
| 4 | **歪みの一貫性**：同一 `ref` は同一の歪み方を維持する（怪物を「気配」と呼んだら最後まで「気配」系で通す。正体が段階的に**明かされていく**のは演出として可。無秩序に揺れるのは不可） | `ref` の同一性（`docs/07`）が意味を失い、LLM が対象を追えなくなる |
| 5 | **不変条件 #1 の骨抜き禁止**：歪みは数値露出の代替手段ではない。「歪んだ数値」も数値（「たぶん3メートルくらい」も禁止）。`assertNoRawMechanics` は従来通り最終ゲート | 不変条件 #1 崩壊 |

> 覚え方：**嘘をつけるのは「見え方」だけ。「選べること」「起きたこと」「向き」には嘘をつかない。**

補足——hard rule 2 と「こわれの前触れ」（`docs/01` 型3）は両立する：press 前の警告が `strange`、押した結果が `bad` なのは**真実の反映**であって歪みではない。歪みが許されるのは、その真実を**どんな言葉で見せるか**の側。

## 4. 視聴者側の前提 — Tier B は「真実」を描く

歪みが劇的アイロニーとして成立するには、**視聴者が真実を知っている**ことが前提：

- Tier B（`docs/11 §2`：リプレイ再生した RawState を描く）は、**AIちゃんの知覚の再現ではなく、視聴者向けの真実**を描く。怪物は怪物として、罠は罠として絵にする。
- Tier A（perception 入力）しか無いゲームでは、歪みは視聴者にも見えない＝アイロニーが立たない。**歪みを主レバーにするゲームは Tier B 前提**で起案する（`docs/03 §5` テンプレの「情報非対称」欄）。
- 画面に数値 HUD を出さないのは従来通り（`docs/11` 原則#5）。**真実を描く＝数値を描く、ではない**。

## 5. 歪み対応表の義務（完了ゲートの拡張）

各ゲームは perception.ts 冒頭コメントの「量→質の対応表」（`docs/07` 完了ゲートの要請・praise-room 慣行）を**3列に拡張**する：

```
RawState の量 → 視聴者が見るもの（Tier B）→ AIちゃんに見えるもの（描写）
例：threat 0..4 → 怪物が段階的に接近する絵 → 「後ろが、すこし重たい気がする」（salience faint→vivid）
```

歪みを使わないゲーム（真実＝描写）は従来の2列のままでよい。その場合も「歪みなし」と冒頭コメントに明記する。

## 6. テスト規範

- `assertNoRawMechanics(perceive(state))`：従来通り最終ゲート。歪んだ描写にも数値・座標・秒が混ざらないこと（hard rule 5）。
- replay 決定論テスト（同 seed＋同 action 列 → perceive deep-equal）：歪みは純関数の一部なので、歪み込みで従来のまま通る。
- hard rule 1（affordances 実在性）は既存の「`affordances[].action ⊆ actionVocabulary` かつ `apply` が受理」検証（`docs/07`）が引き続き担う。

---

## 不変条件チェックリスト

1. 歪みはすべて描写語で行い、数値・座標・秒・内部変数名は歪んだ形でも出していない（#1）。
2. affordances は歪めていない：提示＝実行可能（#2・`docs/07` フィールド契約）。
3. 歪みのために schema・response・契約の形を変えていない（#3・#4）。
4. Tier B が描く「真実」の絵にも技術用語・HUD 数値を出していない（#5・`docs/11`）。
5. 歪みは決定論的純関数：乱数・時刻・状態外情報を参照していない（`docs/02` の決定論）。

## 完了ゲートとの関係

`docs/07`＋この doc を読めば、**別の人がゲームの「歪み対応表」（RawState→視聴者→AIちゃんの3列）を独立に書け、hard rules 5つに照らして合否判定できる**こと。
