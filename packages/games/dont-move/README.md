# 一歩も、うごかないで（dont-move）— 起案文書

クィブル型（`docs/techniques/quibble.md`・優先度1位）の第1号。**勝ち回**——「AIが間違うから面白い」ではなく「**AIの考え方が人間と違うから面白い**」回。

> **節目**：これで実装ゲームは 3 本目（praise-room・dont-press-button・dont-move）。
> 以後、`DreamGame` 契約の breaking change は禁止（不変条件 #4・`docs/02`）。

## 1. 技法→ゲーム変換テンプレ（docs/05 §5）

```text
技法名：クィブル
視聴者が抱く疑問：「動くな」と言われて、どうやって出口まで行く？
AIちゃんが行う選択：言われたとおりじっとする／手をのばす／一歩ふみだす／床のレバーに触れる
面白さが発生する瞬間：「そういう意味かよ」——彼女は歩かず、部屋のほうが出口まで歩く
画面で見える結果：部屋ぜんたいが、きしみながら出口までスライドする
AIちゃんの性格が原因になる箇所：ルールを破りたくない律儀さの延長で、破らない解に至る
25秒で説明可能か：可能（0-2s ルール提示／中盤 レバー発見／終盤 部屋が歩く）
同じ型を何本作れるか（量産性）：高い（禁止ルールの数だけ：喋るな・触るな・見るな…）
```

## 2. 起案テンプレ（docs/03 §6）

一文ルール：
> 「一歩も動くな」と部屋に言われたAIちゃんが、動かないまま出口を目指す夢。

技法（1本1技法・docs/05）：
> クィブル。

モチーフ出典（任意）：
> ChatGPT 壁打ちレビューの看板例「一歩も動くな」（`docs/techniques/quibble.md` のゲーム例の種）。文章の転写なし。

視聴者の問い：
> 動くなと言われて、どうする？（普通の解釈＝「出られない」or「我慢比べ」or「結局歩いちゃう」）

情報非対称：
- 視聴者に見えるもの（Tier B が描く真実）：部屋・遠い出口・床のレバー。レバーを引くと部屋ごとスライドする様子
- AIちゃんに見えるもの（perceive の描写）：**歪みなし**（真実＝描写・docs/04 §5）。情報関係は「AIちゃん＞視聴者」——差は事実ではなく**解釈**（抜け穴に気づくか）で作る。レバーは開幕 salience: faint でさりげなく置く（チェーホフの銃の作法）

画面：
- 「うごかないで」と言う部屋・遠くの出口・足もとのレバー。一瞬で分かる三点構成。

actions：
- `lever`「床のレバーに、そっと触れてみる」／`stay`「言われたとおり、じっとしている」／`reach`「うごかないまま、出口へ手をのばす」／`step`「おもいきって、一歩ふみだす」
- 限定列挙4語（不変条件 #2）。視聴者の予想手（stay）も抜け穴の手（lever）も語彙に実在。
- 語彙先頭は lever（mock の既定プレイ＝クィブル満ち。dont-press-button の wait 先頭と同じ慣行）。

間違い（または勝ち筋）の供給源：
- **勝ち回**：抜け穴（クィブル）。「ルールを破りたい」ではなく「ルールを正確に守りたい」律儀さの延長で抜け穴に至るトーン（quibble.md のヘイト管理メモ）。
- 負け経路の理由付け：step＝好奇心の暴発（前触れ警告つき）／stay＝生真面目な従順＋眠気／reach＝抜け穴の失敗形。

失敗の型マッピング（docs/01 §3）：
- `lever`×3 → **満ち（知恵による満ち）**：部屋が出口まで歩く（terminal）
- `step` 1回目＝部屋がかなしそうに軋む警告（strange・前触れ1拍）→ 2回目＝床が破れる → **こわれ（型3）**（terminal・不可逆）
- `reach`×3 → 届かない空振りの連続 → **すれ違い（型4）**（terminal）
- `stay`×4 → まぶたが重くなり出口が霧に溶ける（clear→faint）→ **見失い（型5）**（terminal）
- 尺切れ → **醒め（型2）**（maxTurns）
- 勝ち回の定石どおり：**予想手（stay）は行き詰まり、抜け穴（lever）が満ちに繋がる**（docs/03 §6）。専用 deadend 経路なし（terminal 時の affordances 空はバックストップ）。

面白さ：
- AIちゃんの性格が出る点：「動くな」を文字どおり誠実に守った**結果として**部屋を動かす。破らない・ズルしない・でも人間と解釈が違う。
- 視聴者がコメントしたくなる点：「その手があったか」「いや動いてるだろ部屋がｗ」「わたしは、うごいてない（名言）」——解釈論争がコメントの燃料。

エンディング例：
- 満ち：部屋が出口のまえで止まる。「……ついた。わたしは、うごいてないよ。へやが、あるいてくれたの」
- 失敗（不可逆）：二歩目で床が破れる。「……ごめんね。まって、って言われてたのに」
- 奇妙な終わり：じっとしたまま出口が霧に溶ける。「……どこ、いっちゃったんだろう。うごいてないのに」

最後の一言例：
> 「わたしは、うごいてないと思う。……へやが、あるいてるだけ」

シリーズ適性・記憶接続点（予約欄・1〜2行）：
> 差し出せるもの：「部屋を歩かせた」記憶＝『ルールは破らずに、べつの読み方をする子』という人格の証拠。後続回で禁止ルールに出会ったとき参照できる。設計はしない・記すのみ（docs/08・次Wave）。

フィージビリティ検証：
- 不変条件 #1〜#5 チェック（一覧は CLAUDE.md）：
  - #1 座標・秒・数値の非露出：roomShift 等は salience（faint/clear/vivid）と描写のみに射影。`findRawMechanics` で全ターンゲート（state.test.ts）
  - #2 action 限定列挙：4語固定。抜け穴 lever も語彙内に実在（自由文字列なし）
  - #3 structured output：core の schema/parse に変更なし
  - #4 契約不変：DreamGame の形をそのまま実装。additive 変更もなし
  - #5 技術用語・技法名の非露出：hook・描写・affordance ラベルは夢の言葉のみ（「クィブル」等は表に出ない）
- 歪みなし（docs/04 §5 の「歪みなし」明記パターン）のため hard rules は自明に通る（affordances 実在・質信号誠実・因果非捏造・一貫性・#1維持）
- 決定論：apply は純関数・乱数なし。seed は開幕描写のゆらぎのみ。replay deep-equal をテストで固定

## 3. 判定パネル自己採点（docs/03 §7・4レンズ）

| レンズ | 採点 | 根拠 |
|---|---:|---|
| フック/拡散 | 8.5 | 「動くなと言われた。出口は遠い。どうする？」は2秒で立つ問い。オチ（部屋が歩く）の絵の落差が大きい |
| キャラ適合/固定ファン | 8.5 | 律儀さがそのまま勝因になる（生真面目×眠そうの生真面目側の見せ場）。名言性のある締め。「ルールを別解釈する子」という人格の積み上げ |
| アーキ適合 | 9 | 単一スカラー中心・限定4語・座標不要・閉じ方5種・決定論・Tier B（部屋スライド）が自然 |
| 期待ギャップ | 8 | 問いの即答性◎／行動理由の納得度◎（勝ち筋＝律儀さ）／結果の可視性◎（部屋の移動）。リスクは LLM が lever に気づかない take の比率（Mode B+ 選定と語彙順で対処） |

期待ギャップは最下位でない → 公開作候補として可。

## 4. ヘイト管理チェック（docs/03 §4）

- **勝ち回**——被害者化リスクなし。直近が負け回（アイロニー等）の後に置くと配合が良い。
- アンチパターン非抵触：罠のための罠なし（部屋は本気で「動くな」と言っているだけ）／落ち度の過剰演出なし（step も好奇心として描き、警告1拍を必ず挟む）／救いゼロ連打なし（全終端に受け止めあり）。
- quibble.md の注意事項：**「ズルで勝つ」に見せない**。彼女はルールを出し抜くのではなく、誰よりも正確に守る——そのトーンを take 選定基準に含める。

## 5. Tier B の絵の意味（codeX 引き継ぎ仕様）

render-frame は本パッケージでは未実装（codeX でアセット生成予定）。実装時は dont-press-button の `render-frame.ts`／`render-demo.ts` を雛形に、**リプレイ駆動**（trace に RawState を積まない・docs/11 §2）で：

- **量→絵の対応**：
  - `roomShift` 0..3 → 部屋（壁・床のフレーム）が出口方向へ段階スライド。**AIちゃんの立ち位置は画面内で不動**（「動いてない」ことが絵で証明される＝オチの核）
  - `drowse` → 画面全体が霧がかって出口が薄れる
  - `stepWarned` → 床に細いひび・部屋がかなしげに軋む揺れ
  - `stepped` → 床が破れて暗転
  - レバー → 開幕から画面内に**小さく**置く（チェーホフの銃：見返した視聴者が「最初からあった」と確認できる）
- **可視テキスト**：ダイジェティックな「うごかないで」のみ。数値・座標・カウンタ変数名・技法名は絵に出さない（不変条件 #1/#5。dont-press-button の `render-frame.test.ts` と同型のテストでゲートすること）
- 9:16（1080×1920）・SMIL または CSS アニメ・`ButtonFrame` 相当の 0..1 質的フィールドに正規化してから描く

## 6. 実行方法

```bash
pnpm --filter @dream/dont-move build
node packages/games/dont-move/dist/demo.js [seed]   # mock（鍵不要）。既定プレイ＝クィブル満ち
DREAM_LLM=nim node --env-file=.env packages/games/dont-move/dist/demo.js  # NIM 1本

```

takes-batch は `usedLever`（抜け穴到達）を take ごとに記録する——クィブル型の打率（何本で気づくか）の観測点。
